## 1. 需求分析与功能规划

### 1.1 核心目标用户群体分析

| 用户群体 | 特征描述 | 核心需求 | 优先级 |
|---------|---------|---------|-------|
| **个人投资者** | 有一定编程基础或兴趣，自主研究量化策略 | 策略编写、回测验证、模拟交易 | 高 |
| **量化爱好者/学生** | 学习量化知识，验证学术想法 | 教育模块、示例策略、文档教程 | 中 |
| **机构用户（私募/自营）** | 专业交易团队，有合规要求 | 实盘交易、风控合规、多账户管理 | 极高 |
| **量化策略开发者** | 策略研究者，专注策略研发 | 高性能回测、因子研究、组合优化 | 高 |

**差异化服务策略：**
- 对个人用户：提供免费基础额度，降低使用门槛，通过社区和教程体系提升粘性
- 对机构用户：提供API对接、定制化风控模块、SLA保障，单独定价

### 1.2 PC端与H5端双端使用场景分析

| 场景 | PC端 | H5端 |
|-----|------|------|
| **策略编写** | ✅ 完整IDE支持，多文件管理，代码补全 | ⚠️ 轻量代码编辑，适合简单修改 |
| **回测运行** | ✅ 大规模计算，本地缓存详细结果 | ⚠️ 发起回测，查看简要结果 |
| **实盘监控** | ✅ 多窗口监控，实时数据看板 | ✅ 关键指标监控，告警推送 |
| **移动办公** | ❌ 不适合 | ✅ 随时随地查看持仓、绩效 |
| **日常研究** | ✅ 大屏多图，分析报告生成 | ✅ 快速查看，碎片化使用 |

**双端协同策略：** 采用"PC为中枢，H5为延伸"的设计理念——重度操作在PC完成，移动端专注监控与告警接收，通过用户体系实现数据实时同步。

| 层次 | 推荐选型 | 备选方案 |
|-----|---------|---------|
| **主要语言** | Python 3.11+ | Go（高频交易网关） |
| **Web框架** | FastAPI（异步优先） | 备选：Gin（Go）用于交易网关 |
| **任务队列** | Celery + Redis / APScheduler | 轻量定时任务用APScheduler |

| **API文档** | FastAPI自动生成OpenAPI 3.0 |  |
| **连接池** | SQLAlchemy Pool + asyncpg |  |

**Python优先的理由：** 量化交易领域Python拥有最完善的生态（pandas、numpy、TA-Lib、statsmodels等），策略研究人员的首选语言也是Python，全栈Python可最大化代码复用。

### 2.3 数据库选型

| 用途 | 推荐方案 | 详细说明 |
|-----|---------|---------|
| **关系型数据** | PostgreSQL 16 | 用户数据、策略元数据、订单记录、账户信息 |
| **时序数据** | TimescaleDB（基于PG扩展） | 行情K线、Tick数据，一套数据库解决两类需求 |
| **缓存层** | Redis Cluster | 实时行情缓存、会话管理、限流计数 |
| **文件存储** | MinIO（兼容S3协议） | 策略文件、回测报告、用户上传文件 |
| **搜索** | Elasticsearch 8.x | 全局策略搜索、用户搜索 |
| **消息队列** | Kafka / RocketMQ | 异步订单流、行情分发、回测任务队列 |

**选型理由：**
- TimescaleDB 是 PostgreSQL 的时序扩展，一张表同时满足关系查询和时序查询，不需要维护两套数据库，技术栈统一，运维成本低
- Redis Cluster 保证实时行情数据的低延迟读写
- Kafka 支撑高并发行情分发（一路行情可分发至数千个订阅者）

### 2.5 API接口设计原则

| 维度 | 设计原则 |
|-----|---------|
| **协议** | RESTful（管理接口）+ WebSocket（实时数据）+ gRPC（内部高频通信） |
| **认证** | JWT Token + RefreshToken，机构用户支持API Key |
| **版本控制** | URL路径版本（`/api/v1/`），保持向后兼容 |
| **限流** | Redis计数器，滑动窗口算法，按用户/接口维度限流 |
| **统一响应** | `{ "code": 0, "data": {...}, "msg": "success" }` |
| **错误码** | 全局错误码体系（参考HTTP状态码扩展），错误码文档化 |
| **文档** | OpenAPI 3.0（FastAPI自动生成） + Swagger UI |

### 2.6 双端数据同步与体验一致性策略

| 同步维度 | 实现方案 |
|---------|---------|
| **用户态同步** | 共享JWT，用户在PC修改策略，H5即时可见（基于轮询或WebSocket通知） |
| **实时数据** | 统一WebSocket网关，PC/H5共用同一个行情推送通道 |
| **离线处理** | H5端IndexedDB本地缓存，联网后与服务端对账同步 |
| **代码同步** | 策略代码存储在后端，PC/H5编辑后实时保存（OT协同或Last-Write-Wins） |
| **体验一致性** | 核心组件库PC/H5共用，UI适配层通过响应式布局/条件渲染实现 |

## 3. 数据层设计

### 3.1 行情数据获取

#### 数据源分级策略

| 级别 | 数据源 | 延迟 | 成本 | 适用场景 |
|-----|-------|------|-----|---------|
| **L1（免费/低成本）** | AKShare、efinance（免费接口） | ~500ms | 免费 | 个人用户、历史研究 |
| **L2（中等成本）** | Tushare Pro、聚合数据 | ~100ms | ¥2000-5000/年 | 机构用户、正式回测 |
| **L3（专业级）** | Wind终端、Choice金融终端 | ~10ms | ¥5万+/年 | 专业机构 |
| **L4（极速）** | 交易所直连（CTP/OMD） | <1ms | 交易所对接费用 | 极致低延迟需求 |

**推荐策略：**
- MVP阶段使用 AKShare + Tushare Pro 组合，平衡成本与数据质量
- 机构版接入 Wind 或 Choice
- 自建数据清洗管道，统一数据格式输出

**策略DSL示例语法设计：**

```python
# 策略 DSL 示例（类Python语法）
INIT:
    self.short_window = 10
    self.long_window = 30
    self.buy_signal = None
    self.sell_signal = None

ON_BAR(security, data):
    ma_short = MA(data.close, self.short_window)
    ma_long = MA(data.close, self.long_window)
    
    if ma_short > ma_long and self.buy_signal == False:
        self.buy_signal = True
        ORDER(security, 'BUY', 1000, type='MARKET')
    elif ma_short < ma_long and self.buy_signal == True:
        self.buy_signal = False
        ORDER(security, 'SELL', 1000, type='MARKET')
```

#### 4.1.2 回测引擎架构

**推荐：事件驱动型回测引擎（而非向量化）**

| 对比维度 | 事件驱动 | 向量化 |
|---------|---------|-------|
| 模拟精度 | ✅ 高（逐笔Tick模拟） | ⚠️ 中（基于K线，存在未来函数风险） |
| 执行速度 | ⚠️ 较慢（Python循环） | ✅ 快（NumPy/Pandas向量化） |
| 适合场景 | 策略研发、信号验证 | 批量参数扫描 |
| 代码复杂度 | 高 | 低 |

**事件驱动回测流程：**

```
策略初始化 (on_init)
    ↓
加载历史数据 → 逐Bar/逐Tick回测循环
    ├── 计算因子 (calculate_indicators)
    ├── 信号生成 (generate_signal)
    ├── 订单执行模拟 (simulate_order)
    └── 绩效记录 (record_performance)
    ↓
回测完成 → 生成绩效报告 (generate_report)
```

**关键组件设计：**

```python
# 回测引擎核心伪代码
class BacktestEngine:
    def __init__(self, config: BacktestConfig):
        self.data_feeder = DataFeeder(config.universe, config.start_date, config.end_date)
        self.portfolio = Portfolio(initial_cash=config.initial_cash)
        self.risk_manager = RiskManager(config.risk_rules)
        self.executor = OrderExecutor(execution_mode=config.exec_mode)
        self.collector = PerformanceCollector()
    
    def run(self, strategy: Strategy):
        for tick/bar in self.data_feeder:
            strategy.on_tick/bar(tick)
            
            # 风控前置检查
            self.risk_manager.pre_check(self.portfolio)
            
            # 撮合模拟
            self.executor.fill_orders(self.portfolio, bar)
            
            # 绩效快照
            self.collector.snapshot(self.portfolio, bar)
```

#### 4.1.3 关键技术难点与解决方案

**难点1：因子计算效率**

| 方案 | 适用场景 | 实现方式 |
|-----|---------|---------|
| **NumPy向量化** | 基础技术指标（MA、EMA、RSI） | 一次性计算全量历史数据 |
| **Numba JIT编译** | 复杂自定义因子 | @njit装饰器加速Python循环 |
| **CuPy（GPU）** | 超大规模因子（全市场Tick级别） | GPU显存存储，计算速度10x+ |
| **预计算缓存** | 常用因子（TA-Lib覆盖的） | 回测前预计算并存储TimescaleDB |

**难点2：滚动回测（Walk-Forward）**

```python
# 滚动窗口回测框架
def walk_forward_analysis(strategy, data, train_window, test_window, step):
    """
    train_window: 训练窗口（如252个交易日=1年）
    test_window: 测试窗口（如63个交易日=1季度）
    step: 滚动步长
    """
    for train_end, test_end in sliding_windows(data, train_window, test_window, step):
        train_data = data[:train_end]
        test_data = data[train_end:test_end]
        
        # 在训练集上优化参数
        best_params = optimize(strategy, train_data)
        
        # 在测试集上验证（样本外）
        performance = backtest(strategy.set_params(best_params), test_data)
        
        yield performance, best_params
```

**难点3：样本外测试（Out-of-Sample）**

| 测试类型 | 方法 | 目的 |
|---------|------|------|
| **Train/Test Split** | 70/30分割 | 基础验证 |
| **滚动窗口** | 逐步滚动训练/测试 | 避免过拟合时间依赖 |
| **蒙特卡洛模拟** | 随机子采样收益率序列 | 验证策略稳健性 |
| **bootstrap** | 重抽样法估计不确定性 | 量化估计区间 |

## 6. 总结与风险提示

### 6.1 技术选型总结

| 层级 | 推荐技术栈 | 核心价值 |
|-----|-----------|---------|
| **PC前端** | React + TypeScript + Vite + Monaco + lightweight-charts | 企业级体验、高可维护性 |
| **H5前端** | Taro 4.x（React） + F2/ECharts | 跨平台高效交付 |
| **后端核心** | Python 3.11 + FastAPI + SQLAlchemy | 量化生态完善、高并发友好 |
| **回测引擎** | 事件驱动 + NumPy/Numba + Celery | 高模拟精度、合理性能 |
| **时序存储** | TimescaleDB（一套数据库搞定时序+关系） | 降低运维复杂度 |
| **缓存** | Redis Cluster | 实时行情低延迟 |
| **消息队列** | Kafka | 高并发行情分发 |
| **部署** | Kubernetes + Docker + ArgoCD | 弹性扩缩容、GitOps交付 |

### 6.2 关键成功因素

1. **数据质量是根基：** 数据错误会导致回测结果失真，实盘损失。因此数据清洗管道（标准化、去重、异常检测）必须作为核心模块认真建设。

2. **回测引擎保真度：** 事件驱动回测是"慢但准"的路径，务必注意避免前视偏差（look-ahead bias）、幸存者偏差等经典陷阱。

3. **风控是生命线：** 实盘交易中，一次未受控的下单可能造成无法挽回的损失。风控必须作为独立模块独立部署，与策略逻辑完全解耦。

4. **合规前置设计：** 中国量化交易受到证监会、交易所的多层监管，接入真实账户前需充分了解相关法规（尤其是高频交易报备规则）。

5. **用户体验渐进式：** MVP阶段不要追求功能大而全，聚焦"策略编写→回测验证→绩效查看"核心闭环，快速迭代。

### 6.3 风险提示

| 风险类别 | 具体风险 | 应对措施 |
|---------|---------|---------|
| **数据风险** | 数据源中断、延迟、数据错误 | 多数据源备份、每日对账校验 |
| **技术风险** | K8s集群故障、数据库宕机 | 主从多活架构、灾备演练 |
| **交易风险** | 策略逻辑缺陷、极端行情 | 多层风控、熔断机制、盘后对账 |
| **合规风险** | 违规交易、监管政策变化 | 法务团队审核、合规模块内置 |
| **安全风险** | API Key泄露、DDoS攻击 | 加密存储、限流、WAF防护 |
| **人员风险** | 核心开发人员离职 | 文档化、代码review、交接SOP |
| **市场风险** | 策略失效（市场风格切换） | 持续监控Walk-Forward表现、样本外跟踪 |
